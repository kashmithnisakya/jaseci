- **Add: Optional `profile` on register, new `GET /user/me`, and SSO profile population**: `POST /user/register` accepts an optional `profile: dict` of arbitrary developer-defined fields (firstname, lastname, address, postcode, etc.). Values must be strings, numbers, or booleans, with bounded limits (20 keys, 64 chars per key, 1024 chars per value, 8192 bytes total) and keys restricted to `[a-zA-Z][a-zA-Z0-9_]*` to prevent MongoDB operator/dot-path injection and JS prototype pollution. The `sso` key is reserved and rejected from client-supplied profile data. Profile is stored as a `profile` sub-document on the user record (Mongo and SQLite backends). New `GET /user/me` returns the authenticated user's `{user_id, role, status, identities, profile, created_at, updated_at}` (Bearer-auth, credentials and normalized identity values stripped). SSO providers (Google, GitHub, Apple) now populate `profile.sso.<platform>` from the OAuth response (`display_name`, `first_name`, `last_name`, `picture`); `SSOUserInfo` is extended to carry these fields, and `sso_callback` refreshes them on every login so they stay current without overwriting developer-set fields outside the `sso` namespace.
