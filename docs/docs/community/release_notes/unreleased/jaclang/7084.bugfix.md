- **Fix: `--scale` deploys no longer crash-loop on missing `pymongo`**: the scale runtime's user manager persists to MongoDB by default, but the pod's capability closure only pulled the mongo driver when a `[plugins.scale.database]` backend was spelled out. A scale app that served without declaring one shipped pods that booted, ran `jac install`, then crash-looped at first request with `pymongo required`. A scale-enabled serve now defaults `data.mongo` into its runtime closure (so `jac install` - host or in-pod - installs pymongo), unless the project names a different `[database]` backend. `--experimental` bundle keeps scale capability intents**: The bundle sanitizer no longer strips the entire `[plugins.scale.*]` namespace from the pod's `jac.toml`. It now removes only the secrets sub-table, in both supported forms (`[plugins.scale.secrets]` and `[scale.secrets]`), so the capability intents (database, storage, monitoring, tracing, ...) survive and the pod's `jac install` can materialize the config-driven runtime closure instead of crash-looping with `Capability 'data.mongo' is required but not installed`. `jac install` venvs ship a build backend**: Newly created `.jac/venv` environments are now seeded with `pip`, `setuptools`, and `wheel` immediately after creation. Previously, ensurepip on Python 3.12+ left the venv without `setuptools`, so installing any source-only (sdist) dependency failed with `Cannot import 'setuptools.build_meta'` unless a caller had pre-installed it by hand. `jac install` is now self-sufficient out of the box.The observability `prometheus_client` module imported `requests` at module level, pulling it into the serve import chain even though it is outside the base serve closure. In pods without `requests`, `server.serve` failed to import, `get_api_server_class` silently swallowed the error and fell back to the core server, and the fallback then died with a misleading `JFastApiServer ... does not support the context manager protocol`. `requests` is now imported lazily inside the Prometheus query functions (and a genuinely missing `requests` degrades to `None`), and `get_api_server_class` logs which module was missing instead of swallowing the error.The shared bundle PVC requests `ReadWriteMany`, but its storage class defaulted to the cluster default StorageClass, which on most EKS clusters (gp3 / `ebs.csi.aws.com`) is `ReadWriteOnce` and cannot provision RWX. The deploy then hung until the bundle-loader pod timed out with an opaque message. `build_bundle_pvc` now fails fast with an actionable error naming the PVC, the access mode, and the fix (set `bundle_storage_class` to a RWX-capable StorageClass such as `efs-sc`).
