- **One trailer owner; `--as binary` is a `.jab` overlay**: the single-binary trailer format now lives only in the Zig launcher (`runtime.zig`) instead of being hand-duplicated in Python (`_split_jac_binary`, the desktop-fuse parser). `jac build --as binary` no longer unpacks and repacks the bundled CPython payload -- it appends the same sealed `.jab` that `--as jab` emits as an overlay onto a copy of the running `jac` (via the internal `jac __appjab`/`__graftrt`/`__hasruntime` launcher verbs), so the artifact is byte-for-byte reproducible. At boot the launcher hands the appended image to `cli_boot`, which mounts it through the same `run_jab_image` path as `jac run app.jab`, reading the entry from the sealed manifest. The `jac_app.json`/`APP_MARKER` boot marker and the `main`/`app`/`__main__` entry auto-probe are gone: a `--as binary` app must resolve an entry the same way `jac run` does (a `main.jac`, or `[project] entry-point`), and an entry-less package is now rejected at build time rather than at first run.
