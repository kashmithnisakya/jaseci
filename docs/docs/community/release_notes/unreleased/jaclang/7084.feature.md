- **Feature: k8s deploy ships a prebuilt `jac` binary, driven by `[dev]`**: the deploy no longer builds jaclang (host or in-pod). It always downloads a prebuilt, self-contained binary from jaseci-labs releases and ships it to pods over the bundle PVC. The project's `[dev]` jac.toml stanza picks the runtime: no `[dev]` runs the latest **stable** release as-is; a bare `[dev]` runs the rolling **dev** prerelease (built from `main` HEAD) as-is - no source overlay, because it already is `main`. Overlaying a jaclang source tree only happens when you request a custom source: `[dev] repo`/`branch`, or `[dev] jaclang_source` pointing at a local `jac/` checkout (else fetched `main`). The overlay is applied via the editable dev loop and the pod compiles it once on first boot. scale and the MCP server are built into the binary, so pods never clone, install plugins, or build.
- **Feature: `--scale` builds the app client on the host and ships it**: the gateway pod no longer runs `jac build` at boot to produce the client bundle. The deploy builds it on the host (where the native/wasm toolchain lives) and ships `.jac/client/dist` over the bundle PVC for the gateway to serve directly. Apps whose entry has a native `na {}` (wasm) block now serve at `/` even though the slim pod binary can't compile them; the in-pod build remains a fallback when no dist was shipped.
- **Static-musl Linux binaries from `nacompile`**: self-contained Linux executables now link fully static against a vendored musl runtime, with no glibc or dynamic-loader dependency, so they run on Alpine, scratch containers, and any glibc version. Executables that import an external shared library (for example a desktop `libwebview` host or an embedded `libpython`) stay dynamic, and the shared-library (`.so`) path is unchanged. The musl runtime is harvested from the bundled Zig toolchain via a new `zig build vendor-musl` step and bundled into the shipped binary.
