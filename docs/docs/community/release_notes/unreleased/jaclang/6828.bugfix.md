- **Fix: macOS/Windows shared-library exports resolve through `ctypes`/`dlopen`**: (macOS) the native linker no longer emits an `LC_LOAD_DYLINKER` load command in Mach-O dylibs (it is only valid for executables and could cause `dlopen` to reject the file) and defaults `CS_EXECSEG_MAIN_BINARY`'s `execSegFlags` to 0 for dylibs (the `0x1` main-binary flag is only valid on an executable's `__TEXT` segment and trips a macOS load error when set on a shared library). (Windows) the per-symbol `clib_symbol_lib` map now propagates transitively from imported `.na.jac` sub-modules to the entry module's import-table generation: previously only `clib_paths` crossed the module boundary while `clib_symbol_lib` was dropped, so every Win32 API symbol imported from a sub-module fell back to `needed_libs[0]` (`msvcrt.dll`) and the loader raised `WinError 127` at runtime. Single-file programs were always correct (the symbol-to-DLL map is built directly on the entry module during IR gen, no propagation needed). Added Mach-O `dlopen`/`dlsym` round-trip and export-trie naming regression tests (macOS-gated), a single-file `clib_symbol_lib` attribution guard, and the existing structural load-command guards.
